About My ELO Project

Match History

How to Use

Enter a Player ID to view their match history, Elo rating, and leaderboard position.

Description

By submitting a player ID that is recognized in the database, you can see that players info (ID, Position, and ELO) and their match history.

Authors' Notes

I was quite sad about myself for this section, as I hadn't figured out how to make it work with Elo Change and Opponent's Elo until today. However, I realized that I was working with data that was made before I started tracking those numbers. Thankfully, this section shows the opponents' elo at the time of the match AND the elo change resulting from the match

Leaderboard

How to Use

View the full leaderboard with the button at the top. You may also use the form to search by Player ID, position, or Elo rating.

Description

This is the segmented display of data needed in the B- requirements. Shows players based off who has the highest ELO rating from the matches reported in the Match Report Form.

Authors' Notes

I'm quite happy with this section! It does basically everything I set out for it to do, and the coding for it made sense both Lambda and front end wise. Setting up filtering was pretty difficult, but it feels like a very important concept, so I'm happy it is implemented.

Match Report

How to Use

Fill out the form to report a new match, including scores, players, and game details.

Description

By filling out the form, users will change the ELO of their player and their opponent according to the ELO algortihm. Their new elos will be displayed as an alert, as well.

Authors' Notes

This was the first thing I did for the project. It was relatively easy, and is adapted from LAB 11. Preferably, this would be the only way for users to input stuff into the database, but index is required...

Index

Description

This is a filler section required by the C- requirements. It is index.html, can add to AWS items database, load all the tables relevant to the project in one button, and has a way to delete "Item" table entries.

Setting this up was more annoying than I thought it would be, since I needed to use Promise.all to make the button not janky... Overall only here for requirements, doing everything in the C- section.

Note on Utilization of Generative AI (more importantly HOW)

ChatGPT and Copilot were used intermittendly when creating this project. They were not used to create code, nor document anything (as my commenting style likely indicates, lol) but mainly as a debugging helper/searcher throughout the process. All code provided within the project is written by me and not generative AI. An example use-case for it would be this: I was having a problem in index where the tables would load in an essentially random order, or not at all. I was trying to look up ways to fix it from our class resources / online but was getting nowhere. Therefore, I asked coPilot to tell me ways to solve this issue. It reccommended a couple solutions, including using Promise.all(). Therefore, I looked up how to use Promise.all() and implemented it. It is embarassing to talk about this, admittedly, but I feel it poignant to state that they were used. I feel like I used them in a relatively healthy manner, but overall at the end I am still a bit bitter about including them in my workflow. However, the truth is I used them, so I am stating it here
